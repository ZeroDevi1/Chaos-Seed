# Rust + Slint 实战检查清单与反模式

## 交付前检查清单

- `AppTheme` 是否是唯一视觉令牌来源
- 自定义组件是否复用了统一的颜色、边框、按钮状态
- `std-widgets` 是否与 `Palette` 同步
- 列表模型是否保持稳定实例，只更新内容
- Rust callback 是否只做轻量工作
- 后台任务是否通过统一消息通道回 UI
- 子窗口创建时是否同步主题、调色板、尺寸和图标
- 透明或 Overlay 窗口是否也同步主题
- 长生命周期 `Timer` 是否明确保活或释放

## 常见反模式

### 颜色散落

在多个 view 里直接写颜色常量，最终会让浅色/深色切换和品牌色调整变成全局返工。

### 把业务写进 .slint

在 `.slint` 里堆解析、下载、网络请求或复杂状态机会让 UI 文件变脆，也让测试与重用更困难。

### 反复替换模型对象

列表刷新时频繁替换 `ModelRc` / `VecModel` 指针，容易带来代理不刷新或滚动状态丢失等问题。

### 忘记同步 std-widgets 调色板

自定义壳层切到深色后，输入框仍是浅色，这是 Slint 混用方案里最常见的割裂点。

### 子窗口像独立应用

聊天窗、Overlay、浮窗各自定义颜色和主题规则，会让整套应用失去统一感，也增加维护成本。

## 排查顺序

1. 先看主题源是不是只有一套。
2. 再看当前控件是不是选错了技术路线。
3. 再看 Rust 是否真的把状态同步到了全局对象和子窗口。
4. 最后排查模型、定时器和消息通道。
