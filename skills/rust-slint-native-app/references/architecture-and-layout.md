# Rust + Slint 架构与目录模式

## 目标

为 Rust + Slint Native 应用建立可扩展的 UI 分层，让“主题、组件、页面、窗口、Rust 绑定”各有边界。

## 推荐目录

```text
src/
  main.rs
  ui/
    app.slint
    theme.slint
    models.slint
    components/
    views/
    windows/
```

## 分层职责

- `theme.slint`
  - 只定义全局视觉令牌和主题状态。
  - 不写页面布局。
- `models.slint`
  - 只放共享 UI 数据结构。
  - 适合表格行、列表项、窗口共享的展示模型。
- `components/`
  - 放可复用视觉原语。
  - 典型内容：按钮、侧栏、字段标签、卡片、状态条。
- `views/`
  - 放主窗口中的页面。
  - 页面负责拼装组件与绑定属性，不负责后台线程或窗口创建。
- `windows/`
  - 放聊天窗、Overlay、浮窗之类的独立窗口。
  - 独立窗口优先单独建文件，不要通过主页面条件分支伪装。
- `main.rs` 或专门的 Rust UI 控制器
  - 创建窗口。
  - 安装 callbacks。
  - 管理模型、消息泵、后台任务和主题同步。

## 推荐构建顺序

1. 先定义 `AppTheme` 和基础令牌。
2. 再定义基础组件。
3. 再定义主窗口和页面切换骨架。
4. 最后接 Rust 的 callback、模型和后台任务。

## Rust 绑定模式

- 用属性做“当前状态”。
- 用 callback 做“用户动作”。
- 用 `VecModel` / `ModelRc` 做“列表数据”。
- 用统一消息通道把后台结果投递回 UI。

## 多窗口模式

- 主窗口负责持有全局设置和主题状态。
- 子窗口创建时同步主题、调色板、尺寸和图标。
- 子窗口如果共享列表数据，优先共享稳定模型实例，而不是复制状态。

## 文件组织红线

- 不要在 `theme.slint` 混入页面逻辑。
- 不要在 `.slint` 中塞后台任务与耗时流程。
- 不要让多个 view 各自维护一套颜色、边距和状态定义。
