# 接入路线选择

## 快速判断

先决定“谁拥有逻辑”，再决定“怎么暴露逻辑”。
如果逻辑还没放对层，就不要先争论 FFI 还是 daemon。

## 选择矩阵

| 路线 | 适合场景 | 不适合场景 | Chaos-Seed 示例 |
| --- | --- | --- | --- |
| in-process | Rust UI 壳、调用链短、类型共享方便、无需跨语言 | 需要独立进程隔离、统一后台、跨语言接入 | `chaos-slint`、当前 `chaos-tauri` |
| FFI | 跨语言直嵌、同步调用多、调用面窄、宿主可接受加载成本 | 需要流式通知、会话共享、多客户端状态 | `chaos-ffi`，WinUI3/Flutter 可选 |
| daemon | 独立后台、鉴权、通知推送、可重启、共享任务和状态 | 只做少量同步调用、协议维护成本不值得 | `chaos-daemon` + WinUI3 |

## 默认策略

- Slint、Tauri 这类 Rust 壳默认优先进程内
- WinUI3、Qt、C#、Flutter Windows 先在 FFI 与 daemon 中二选一
- Android 或嵌入式宿主如果只需少量核心能力，优先考虑 FFI 或更轻的共享 crate，而不是搬整套 daemon

## 什么时候选 FFI

- 能力以请求/返回为主
- 不需要统一后台任务中心
- 不需要复杂鉴权和通知流
- 宿主语言已有自己的 UI 状态管理，只需要调用 Rust 能力
- 你愿意明确处理字符串编码、内存释放、线程切换和回调 marshal

## 什么时候选 daemon

- 需要把会话、下载队列、订阅、弹幕流、语音流放进独立后台
- 需要服务器通知、流式事件、轮询状态、鉴权或重连
- 需要让多个客户端共享同一份状态或后台任务
- 需要把崩溃隔离在后端进程，而不是拖垮宿主 UI

## 什么时候混用

- 同一个壳既有高频、同步、轻量能力，也有长生命周期后台任务
- 例如：配置读取、简单解析走 FFI；下载队列、直播订阅、流式通知走 daemon
- 混用时仍要保持 DTO、错误语义和能力源头一致，不要一边一个协议

## 落地提醒

- 先把共享形状收敛到独立 `proto` crate 或明确的 JSON 约定
- 让 `ffi` 只适配 ABI，不要偷放业务状态机
- 让 `daemon` 只适配传输，不要复制核心算法
- 如果未来还要给 MCU 复用，优先继续往下抽共享 crate，而不是让 FFI/daemon 成为唯一入口
