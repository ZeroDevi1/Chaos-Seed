# Rust + Tauri + Svelte 实战检查清单与反模式

## 交付前检查清单

- 是否在挂载前应用了早期主题
- `html[data-theme]` 是否是壳层主题入口
- Fluent token 是否只在必要时写入，且避免重复写入
- 是否只注册了实际使用的 Fluent 组件
- `style.css` 是否承担全局 token 和壳层样式，而不是页面各自造变量
- `shared/` 是否收敛了 `invoke`、`listen`、DTO 和窗口解析
- 多窗口是否有清晰的 `BootView` / label / 入口映射
- Overlay、Player、Dock 等特殊窗口是否有独立背景策略
- store 是否区分了偏好状态、窗口存在性和页面局部状态
- Rust command 的参数、返回和命名是否清晰稳定

## 常见反模式

### 把 Fluent 当整站样式系统

Fluent 更适合控件层，不适合直接承担整套应用壳层。把布局、卡片、结果页全塞进 Fluent，通常会失去桌面应用的完整感。

### 每个页面自己管主题

页面组件各自读写 `data-theme`、各自处理系统主题，会让多窗口和切页行为变得不可预测。

### 窗口分类写死在入口文件里

当聊天窗、Overlay、Player 越来越多时，把判断逻辑堆在 `main.ts` 会迅速失控。应尽早抽成独立窗口解析模块。

### 到处直接 invoke

页面组件直接四处 `invoke` 会让桥接协议、错误处理和测试都碎片化。

### 忽视二级窗口闪白

主窗口看起来正常，不代表聊天窗、浮窗、Overlay 也正常。没有早期主题时，这些窗口最容易先闪一帧白底。

## 排查顺序

1. 先看主题入口是否只有一套。
2. 再看 Fluent 和 CSS 的分工是否混乱。
3. 再看窗口解析与入口映射是否清晰。
4. 最后排查 store、bridge 和 Rust command。
